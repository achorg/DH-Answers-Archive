  4. Bob DuCharme

    Replying to @Bethany Nowviskie's post:

    "Gratuitous violence" is itself a bit gratuitous. I'm sure Eric is aware of SKOS, which uses RDF to represent hierarchies to great advantage.

    Since the RDF you have now links to information that lets people access the TOC, I'm not sure what value it would add to represent the TOC in the RDF, unless you want to store metadata about each individual TOC entry in the RDF. I didn't see an XML representation of the book at http://openlibrary.org/books/OL7173600M/The_log_of_a_cowboy outside of delivery format XML in the DAISY and the ePub, but if you made something like a TEI version available and you had extra metadata about each chapter heading, you could store it right in there using RDFa, which isn't strictly for HTML (see http://www.devx.com/semantic/Article/42543).

    I think the bottom line is that you currently offer a machine-readable way for people or systems to access all the information you want to make available, including the TOC entries, so unless you have new (meta)data you want to also provide, in which case RDFa or more data in your OL7173600M.rdf file would be something to consider, you're fine right now.

  6. Patrick Murray-John

    It might be that the question comes around to vocabularies that are good for representing a TOC. And, as Bob mentions, that might involve new metadata, possibly metadata that gets at the part-to-whole relationships of the contents to the book as a whole.

    If you have metadata about each of the parts/chapters, that could conceivably be done with dct:hasPart between the book itself and the chapters. That wouldn't automatically represent the order, however, which would clearly be a good feature in a TOC.

    I see you are using the Bibliographic Ontology for author data. You might be able to use its chapter classes and properties for the TOC. It gives classes for bibo:Book and bibo:chapter. I think it reuses dct:hasPart to related them, and if I remember right bibo:chapter can be used to show the chapter number. So you might have

    <http://openlibrary.org/books/OL7173600M> a bibo:Book ;
      dcterms:hasPart ex:OL7173600M_1 . 

ex:OL7173600M_1 a bibo:Chapter ;
      bibo:chapter "1";
      bibo:pageStart "1";
      dcterms:title "Up the Trail" .

    At least that's my first guess about how to model it. Page numbers might be a complication, since there you might be dealing with how it gets represented in the output formats.

    An alternative is to go FRBR with it. I'm not sure that that'd quite be right for your examples, though.

    Hope that helps!

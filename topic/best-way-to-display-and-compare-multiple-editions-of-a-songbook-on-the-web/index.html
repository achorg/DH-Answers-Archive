    The most complete way to represent common Western music notation digitally is with the MusicXML format:

    http://www.recordare.com/musicxml

    This format is supported by over 140 programs. If the songbook was created in the past 10 years, it most likely was created by a program like Finale or Sibelius that can export MusicXML files as well as create the PDF files.

    There are a variety of web-based tools for displaying and playing MusicXML files within a web browser, including the free Myriad music plug-in, the Noteflight products, and the enterprise-strength Legato Sheet Music Viewer. Just being able to display and play music notation in the web is challenging enough. I don't know of any product at this point that can do differences in the way that musicians would expect to see them. Sure, you can use an XML differencing tool on the MusicXML files, but I don't think that would be very helpful for musicians.

    Maybe some of the analysis toolkits using MusicXML could help, such as MIT's music21 software? Perhaps it could be hooked up with Juxta or other types of differencing tools?

    You can find links to these different MusicXML software tools and lots more at:

    http://www.recordare.com/musicxml/community/software

    It would help to know more about the source material. What specific music are we talking about? What is the instrumentation? Are there digital source files available besides the PDF image files?

    Good luck with your project!

    Michael

  7. Joanna Swafford
    Member

    Hi Miriam,
    If you want to have any sort of markup, then I'd second Bethany's recommendation for MEI (Music Encoding Initiative). It's an XML music markup based on TEI standards that lets users track scholarly information as opposed to just formatting information (which is all that MusicXML can do). I think people can use it to track differences between editions, and it can also be used for more specialized notation. It was created by Perry Roland at UVa, and he's been very helpful answering all the questions I've had.

    Here's the website for more information about MEI: http://www2.lib.virginia.edu/innovation/mei/
    Here's a tutorial if you want to learn how to write MEI: http://music-encoding.org/documentation/tutorial

    Obviously, using MEI would be more involved than putting some PDFs online and letting users spot the differences themselves, but it's a great option if you have the time.

    Hope this helps, and good luck!

    Hi Miriam,

    MEI does indeed support shape-note notation.

    It also allows the comparison of image regions without encoding the entire group of songs. In the following example --


    <facsimile>
    <surface>
    <graphic xml:id="g1" xlink:href="A_1.jpg"/>
    <zone xml:id="g1z175" ulx="120" uly="992" lrx="450" lry="290"/>
    </surface>
    <surface>
    <graphic xml:id="g1" xlink:href="A_2.jpg"/>
    <zone xml:id="g1z175" ulx="120" uly="992" lrx="450" lry="290"/>
    </surface>
    </facsimile>

    surface elements are used for each page, graphic elements declare the image associated with the page, and zone elements define the regions of interest of the image. The attributes ulx, uly, lrx, and lry (upper-left-x, upper-left-y, lower-right-x, and lower-right-y, respectively) capture the image region's coordinates using the coordinate system of the image, similar to the way image maps are done. The @decls attribute is available on facsimile, surface, and graphic elements in order to associate these elements with metadata in the file header.

    I don't believe this will work for PDFs so you'd have to convert them to some raster format.

    However, having done this, features encoded in the markup (measures, notes, etc.) can be associated with one or more zones using the @facs attribute.


    <measure xml:id="m2" n="2" facs="g1z175 someOtherRegion"/>

    If the differences are all at the measure level, then this markup may be all you need. This provides enough information to locate, in this case, measure 2 in all the appropriate images and navigate between the various images and regions.

    If you are interested in providing commentary about the markup features, then use the annot (annotation) element, like so --


    <annot plist="m2">Measure 2 is really cool!</annot>

    The @plist attribute contains the ID of the feature you want to comment on.

    This is just a basic introduction. If you'd like more info, feel free to write to me at info (AT) music-encoding (DOT) org.

    Good luck with your project!

    People interested in this thread might want also to take a look at Aruspix, which is software for editors of early music prints. Aruspix does optical recognition of music and text, and has features to help with superimposition of images and collation of different editions. I have never used it and don't know the current state of its development, but saw an impressive presentation about it a year and a half ago. The optical music recognition (OMR) features at the time were specifically tuned to particular styles of early music printing, and it's more of a desktop tool for editors than a web presentation tool, so it's probably not directly suited to Miriam's question.

    Hi Miriam,

    MusicXML handles shape-note music as well, and has working software to exchange shape-note music between multiple programs that musicians use.

    MEI has a lot of good ideas, but there is little or no software that uses it. This makes it difficult to use for practical projects such as web displays. It also restricts the usage of your markup to a small community that excludes performing musicians. Any performing musician who uses a notation program can use MusicXML files.

    If you are on a tight timeline, image display of examples does seem to be your best bet. Good luck with your project!

    Michael

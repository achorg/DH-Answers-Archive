You'll see this limitation for XSLT 2.0 transforms, you're limited to using Saxon and it's supported libraries (Java and .Net). Really you're really left with two options if you want to use a functional language in your app, you're kind of limited to either jRuby (or something funky like LIFT or Clojure) or pre-processing the partial views. </p> <p>That said, there is a partial C library for the XSLT 2 spec called libX (<a href="http://www.explain.com.au/libx/)" rel="nofollow">http://www.explain.com.au/libx/)</a>. If this project takes off, you could very easily write a wrapper in PECL, Ruby, or Swig to expose this functionality to the different languages. But it's not there yet. </p> <p>What I would encourage you to also look at is some automated workflows. There are some heavy-weight options (e.g. BPEL), but also some pretty light options (see <a href="http://github.com/jmettraux/ruote)" rel="nofollow">http://github.com/jmettraux/ruote)</a> that go a long way in helping you establish a sane, sustainable, and ultimately transferable method of ingesting and disseminating content, independent of what front-end technology is chosen. </p> knickum.unl@gmail.com on "Alternatives to Cocoon?" http://digitalhumanities.org/answers/topic/alternatives-to-cocoon#post-392 Fri, 08 Oct 2010 17:57:18 +0000 knickum.unl@gmail.com 392@http://digitalhumanities.org/answers/ <p>Been meaning to respond to this also, co-worker of Karin's.
